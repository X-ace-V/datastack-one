# DataStack One — Agent operating rules & accumulated lessons

Local-first "Vercel for data platforms" MVP, built for a hackathon.
This file is read at the start of **every** build session (the standard `AGENTS.md`
convention). Keep it short — standing rules on top, one-line lessons appended below.

Authoritative references: `PRD.md` (contract) · `ARCHITECTURE.md` (design) ·
`TASKS.md` (queue) · `LOOP.md` (how the loop operates) · `PROGRESS.md` (log).

## Build / run

- `npm test` (vitest, full suite) · `npm run typecheck` (tsc) · `npm run dev` (server + web).
- Backend Fastify on `:3001` embeds the OpenCode server; web (Vite) on `:5173`.
- Secrets in `.env` (gitignored): `ANTHROPIC_API_KEY` and other provider keys — never commit.
- `data/` is gitignored (warehouse.duckdb, landing/, uploads/, artifacts/). `fixtures/`
  IS committed (synthetic data only).

## Hard rules (never violate)

- **Definition of Done and acceptance live in `LOOP.md`. A task is not done until every
  gate there is green.** No stubs, no placeholders, no `TODO`, no "simple for now."
- **Never make a gate pass by weakening it** — no deleting/skipping/`.only`/`.skip`ing
  tests, no lowering assertions, no commenting out checks.
- Commit style per `LOOP.md`: conventional, ≤72-char imperative subject, body says WHY.
  **No trailers. No Co-Authored-By. No AI attribution anywhere.** Developer-authored style.
- TS strict, ESM NodeNext — relative imports end in `.js` even in `.ts`.
- `server/core` is pure (no fs/net/process); routes/tools/opencode import core, never reverse.
- One task per iteration; whole suite + typecheck green before check-off.
- Every write/execute tool (`land_parquet`, `load_warehouse`, `run_transform`,
  `publish_serving`) is permission `ask` — never downgrade one to `allow`.
- Synthetic fixtures only; never pipe real data through a free model.
- Default model is `opencode/big-pickle` (free). Keep everything model-agnostic.
- **Database credentials never reach the model.** Connection URLs are entered only in the
  Settings → Connections panel and stored server-side (gitignored). Agent tools take a source
  **name**; the backend resolves name→URL for `ATTACH … READ_ONLY`. A raw URL/secret must
  never appear in a prompt, a tool argument the model produces, an SSE event, or the repo.

## Lessons learned (append, never repeat)

- `TASKS.md`/`PROGRESS.md`/`LOOP.md`/`loop.sh` are gitignored (internal loop state) — edits to them won't appear in `git status`/`git add`; commits contain code only. That's by design, not a bug.
- The `.js`-extension import rule is server-only (NodeNext). `web/` has its own `web/tsconfig.json` (`moduleResolution: bundler`, `jsx: react-jsx`, DOM lib) and uses **extensionless** relative imports; `npm run typecheck` runs both projects. Web component tests live at `web/**/*.test.tsx`, opt into jsdom via a `// @vitest-environment jsdom` docblock, and rely on `@vitejs/plugin-react` in `vitest.config.ts` for JSX.
- To smoke-run ESM server modules ad hoc, write a temp `.mts` file and `npx tsx path.mts` — `tsx --eval`/`-e` compiles to CJS and rejects top-level `await` ("not supported with cjs output format"). DuckDB values come back with bigint fields (e.g. counts), so JSON.stringify them with a bigint replacer.
- `@opencode-ai/sdk` 1.18.3's typed `Config.permission` only accepts fixed built-in keys (`edit`/`bash`/`webfetch`/`doom_loop`/`external_directory`) — it has NO `"*"` wildcard and NO per-custom-tool keys, so ARCHITECTURE §6's literal `{ "*": "allow", "land_parquet": "ask", … }` does not typecheck. Custom write/execute tools are gated at the permission-event layer (the `permission.asked` event + `client.postSessionIdPermissionsPermissionId` reply endpoint, T1.4), not in static config. `server/opencode/config.ts` exports `ASK_TOOLS` as that gate's source of truth and sets the built-in mutation surfaces to `ask` as defense-in-depth.
- `createOpencode`/`createOpencodeServer` spawn the real `opencode` binary via cross-spawn — it must be on PATH, so `opencode-ai` is a runtime dependency (its bin is symlinked into `node_modules/.bin/opencode`). Booting is offline and needs no provider auth as long as you only hit `config.get()`/`app.agents()` and never prompt a model. The SDK's 5s boot timeout is tight for a cold start; the wrapper defaults to 30s. The client returns `{ data, error }` per call — assert `error` is undefined, then read `data`.
- `client.event.subscribe()` returns `{ stream }` directly (an `AsyncGenerator<Event>` from the SSE core), NOT the `{ data, error }` envelope that `config.providers()`/`session.create()` return — iterate it with `for await`. Pass `{ signal }` to abort the pump on shutdown. Event `sessionID` lives in different places by type: directly on `properties.sessionID`, under `properties.info` for `message.updated`, under `properties.part` for `message.part.updated`. Fastify `app.inject` buffers a long-lived SSE response forever, so test the streaming path over a real socket (`app.listen({port:0})` + `fetch` + read the body stream); use `reply.hijack()` + `reply.raw.writeHead/write` and unsubscribe on `req.raw.on("close")`.
- The approval gate (T1.4) rests on SDK names that differ from PRD/ARCHITECTURE prose: the runtime raises `permission.updated` (NOT `permission.asked`), whose `properties` is a `Permission` `{id,type,pattern?,sessionID,messageID,callID?,title,metadata,time:{created}}`; the id is the `:requestID`. Reply via `client.postSessionIdPermissionsPermissionId({path:{id:sessionID,permissionID},body:{response:"once"|"always"|"reject"}})` (NOT `client.permission.reply`), which returns `{data,error}` — a bad session yields an `error` envelope (NotFoundError), never a throw. The MVP maps approve→`once`, reject→`reject`, and never sends `always` (FR8 needs per-call approval). The gate is fed events from the run bridge's single `onEvent` pump so `event.subscribe()` is read once; it does NOT open its own subscription.
- Persisting request bodies to DuckDB: bind user input through positional `$1..$n` params — `store.run`/`store.all` take an optional `values` array (`connection.run(sql, values)`) — never string-concatenate. `getRowObjects()` returns `TIMESTAMP` columns as `DuckDBTimestampValue` objects (and counts as `bigint`), so `SELECT CAST(created_at AS VARCHAR) AS created_at` to hand a plain string to a zod schema. App-generated ids via `crypto.randomUUID()`; read the row back after insert so table defaults (warehouse, created_at) are what the caller sees.
- DuckDB `read_csv_auto(?)` accepts the file path as a bound query parameter (including inside `DESCRIBE SELECT * FROM read_csv_auto(?)`), so file-path table-function args need no string-concatenation — bind them like any other value via `store.all(sql, [path])`. Untrusted CSV **header** names still reach SQL as identifiers, so double-quote each (`"${name.replace(/"/g,'""')}"`) before interpolating. `count(DISTINCT col)` excludes NULLs, so a candidate-key test must also require `nullCount === 0`, not just `distinct === rows`.
- `@opencode-ai/sdk`'s `session.prompt` has NO native json_schema / structured-output field — `SessionPromptData.body` is `{ system?, model?, tools?:{[k]:boolean}, parts:[...] }` and the 200 response is `{ info, parts: Part[] }`. So "structured output" for a stage is: instruct the model to emit ONE JSON object, join the response's `type:"text"` parts (empty separator — parts are contiguous chunks, `\n` corrupts a JSON payload split mid-token), then extract first-`{`..last-`}` and validate with zod (tolerate ```json fences + surrounding prose). Split the failure modes: a bad model *output* is a parse error (route → 422), a session/prompt *runtime* error envelope is distinct (route → 502). Disable tools for a reasoning-only stage (`tools:{tool:false}` for every ASK_TOOLS + built-in bash/edit/write/patch/webfetch) so the model can't fire a `permission.asked` nobody answers. The free `opencode/big-pickle` DID complete the plan stage's structured output live (valid ELT/duckdb/6-step plan) — no paid upgrade needed for planning.
- vitest 2.x peers on vite ^5, so keep root `vite` on ^5 — installing vite@6 makes vitest pull a second nested vite and the two `Plugin` types clash in `vitest.config.ts` typecheck.
- DuckDB `COPY (...) TO '<dir>' (FORMAT PARQUET, PARTITION_BY (col), OVERWRITE_OR_IGNORE)` writes a Hive layout `<dir>/col=<value>/data_0.parquet` but does NOT create intermediate parent directories — `mkdir(landingPath, {recursive:true})` before the COPY or it fails with `IO Error: Failed to create directory`. Query params (`?`) DO bind inside the COPY subquery (`CAST(? AS DATE)`, `read_csv_auto(?)`); the `TO` destination cannot be a param, so single-quote-escape it (`'`→`''`). Reading the partitioned Parquet back, use `read_parquet('<dir>/**/*.parquet', hive_partitioning=true)` — but a `min()`/`max()` aggregate over the Hive **partition** column crashes DuckDB 1.5.4 with an INTERNAL statistics-propagation assertion (`ParquetPartitionRowGroup::GetColumnStatistics`), so count rows with a plain `count(*)` and never aggregate the partition column.
- The pipeline runner (T4.4, `server/pipeline/runner.ts`) is **deterministic**, not agent-driven: it executes each stage by calling the tool functions directly (`landParquet`/`loadWarehouse`/`runTransform`) and gates each `ask` tool through its **own** `RunApprovalGate` (`server/pipeline/run-approvals.ts`, `deps.approve`) — NOT through OpenCode's `permission.updated` events / the T1.4 `ApprovalGate`. ARCHITECTURE §4/§6 sketch the agent calling the tools and OpenCode raising permissions, but the built tools are plain fns (never registered as `@opencode-ai/plugin` tools) and the free model can't reliably orchestrate multi-tool runs, so the scripted-pipeline mitigation (PRD §7) governs execution. The two approval gates coexist: the OpenCode one (T1.4) is for the agent-invoked path (unused by the runner); the run gate is the operative FR8 enforcement. To add a stage (T5.1 DQ / T5.2 Publish): append it to `PIPELINE_STAGES` in `core/run.ts` AND add a `stage(...)` block in `runPipeline`; a gated stage wraps its tool call in `gate(step, tool, summary, sql, args)`. The run's progress rides `GET /api/runs/:runId/events` via `bridge.publish(runId, frame)` (the runner's `emit`), a separate channel from the OpenCode event fan-out. `POST /api/projects/:id/run` launches the runner in the background (fire-and-forget via injected `RunLauncher`) and returns 202 — a run outlives its HTTP request because it pauses for human approval; the client answers gates via `POST /api/runs/:runId/approvals/:requestID` and can recover pending gates from `GET /api/runs/:runId`.
- jsdom page tests race the mount's async fetches: `findByRole` finds a button/element the moment it renders, which can be BEFORE the fetch's `.then` flushes the state that enables/populates it. Assert the post-fetch condition inside `await waitFor(() => expect(...))`, never as a bare `expect` right after `findByRole` — the bare form flakes under parallel `npm test` load and can fail deterministically once enough other jsdom tests run concurrently. (This is what bit `Review.test.tsx`'s "approve button enabled" assertion; fixed by wrapping it in `waitFor`.) Separately, when driving a run in a test, `await screen.findByRole(...)` OUTSIDE any `act(async () => …)` wrapper first (let the mount fetch settle), THEN do the state-changing click inside `act` — calling `findByRole` inside the `act` wrapper starves the pending fetch microtask and the element never appears.
- `EventSource` does not exist in jsdom, so a page that opens an SSE stream (the Run page via `subscribeRunEvents`) must be tested with a fake: `vi.stubGlobal("EventSource", FakeEventSource)` where the fake records `addEventListener(type, cb)` per named channel and exposes an `emit(event)` that calls the listeners for `event.kind` with `{ data: JSON.stringify(event) }`. Drive each `emit` inside `await act(async () => …)` since it triggers React state. The backend frames run events as `{ event: RunEvent.kind, data: RunEvent }` (see index.ts `bridge.publish`), so the browser listens on the four named channels `run.status`/`step.status`/`approval.requested`/`approval.resolved`. SSE frames are NOT buffered for late subscribers (`bridge.publish` drops a frame when no sink is connected), so after subscribing, the Run page does a one-shot `GET /api/runs/:runId` to recover any approval that fired before the stream connected — otherwise a fast first gate deadlocks the UI.
- FR7's "DQ failure blocks publish" is realized in the **deterministic runner** (T5.1), not ARCHITECTURE §5's `tool.execute.before` hook — the tools are plain fns, never `@opencode-ai/plugin` tools, so there is no hook to hang it on (see the T4.4 lesson). The `dq` stage runs `run_dq_check` and throws when `result.passed === false`, failing the run so no later stage (Publish, T5.2) can execute. Consequence for stage-adders: `gated === (tool !== null)` is NO LONGER an invariant of `PIPELINE_STAGES` — `dq` names a tool but is ungated because it is read-only (permission `allow`; it only runs SELECT counts and a failure aborts rather than asks). Gated stages are a strict subset of tool-bearing ones; assert the gated set explicitly (`land_parquet`/`load_warehouse`/`run_transform`) rather than deriving it from `tool !== null`. Inside `run_dq_check`, a check whose query throws (e.g. an unknown column) is recorded as a **failed** check carrying the error detail rather than aborting the loop, so one malformed check cannot mask the others and a missing column fails honestly.
- The served output's **row order is not a contract** (T5.2): the generated transform is a `GROUP BY` with no `ORDER BY` and `publish_serving` exports a plain `SELECT *`, so DuckDB's group order decides CSV row order (it emitted `south` before `north`) — assert the header + rows order-independently, and never let the serve route (T5.3) or the `DataTable`/dashboard (T5.4) imply an order the pipeline doesn't guarantee. Two related shapes worth reusing: `PIPELINE_STAGES` is now **complete at six** (Extract→Land→Load→Transform→DQ→Publish) so T5.3+ add routes/UI, not stages; and `platform.served_tables` is a **registry keyed by `name`** (PRIMARY KEY + `ON CONFLICT (name) DO UPDATE`), not an append log like the other tables — the served name IS the endpoint URL, so exactly one table may answer it and a re-publish replaces the row (publish history lives in `runs`/`approvals`, FR12). A gated stage whose SQL is derived (not human-authored like the transform's) must build it **once** — `planPublishServing` is called by both the runner's gate and the tool — so what the human approves is literally what executes, rather than two look-alike strings that can drift.
- The generated serving endpoints (T5.3, `server/serving/reader.ts`) read the **published CSV export**, NOT the live `marts` table — deliberately. The export is the snapshot that passed DQ and that a human approved at the publish gate, so serving it is what keeps FR7 honest: a later run whose transform succeeded but whose DQ *failed* leaves new rows in `marts` and never publishes, and serving `marts` live would hand exactly that un-published, DQ-failed data to REST callers. It also makes `/api/serve/:name` and `/api/serve/:name.csv` the same bytes by construction and makes paging stable (a file has a fixed row order; the transform's `GROUP BY` without `ORDER BY` does not — lesson above). Consequences for T5.4's page: a registry row whose export was deleted is `410 Gone`, not 404 (the name IS still registered) — `data/` is gitignored and disposable, so this is a real state; and `rowCount` is the TOTAL served, not `rows.length` (the route pages, default 100 / max 1000, and rejects an unusable page rather than clamping). Two things worth not re-deriving: Fastify/find-my-way routes `/api/serve/:name.csv` as a **static suffix distinct** from `/api/serve/:name` (`:name` excludes the `.csv`; no manual disambiguation needed, and `safeServedName` already forbids dots), and DuckDB **does** bind `LIMIT ?`/`OFFSET ?` as well as `read_csv_auto(?)`. Finally, every served cell MUST go through `toJsonCell` (`core/serving.ts`): DuckDB returns `BIGINT` as a JS `bigint`, which `JSON.stringify` **throws** on (a transform emitting `count(*)::BIGINT` is enough to hit this live), and `DATE`/`TIMESTAMP`/`DECIMAL` as value objects — a bigint beyond the IEEE-754 safe range is stringified rather than silently rounded to a different number.
- To smoke-render a **web** page ad hoc, `npx tsx --tsconfig web/tsconfig.json smoke.mts` — the ROOT tsconfig sets no `jsx`, so plain `npx tsx` emits classic `React.createElement` and the page dies with "React is not defined"; web/tsconfig.json's `jsx: react-jsx` fixes it. Node 22 defines `globalThis.navigator` as a getter-only property, so jsdom globals must be `Object.defineProperty(globalThis,"navigator",{value,configurable:true})`, not assigned. Worth doing: rendering the real page in jsdom against a real `app.listen({port:0})` backend with fetch UNMOCKED (wrap fetch to prefix relative `/api/*` with the base, as the Vite dev proxy does) exercises page+route+DuckDB together, which the mocked-fetch component tests cannot — it's the honest DoD #6 observation for a UI task, and it reads geometry (e.g. a bar's `style.width`) the text assertions miss.
- A table existing in the T0.4 migration is NOT evidence its feature is wired: `platform.dq_results` was created by the very first migration and **no code ever wrote to it** until T5.5 — the DQ stage only put an aggregate sentence in `run_steps.detail`. Before trusting any schema object, grep for its writers, not just its DDL. FR12's four records live in four different places and only two were persisted before T5.5: steps (`store/runs.ts`, by the runner), approvals (`store/runs.ts`, by the **route** at decision time — so a run driven straight through `deps.approve` in a test records none), and now tool calls + DQ results (`store/lineage.ts`, by the runner). Beware the two different meanings of "approvals" on the same run: `GET /api/runs/:runId` returns the **pending in-memory** gate requests (live-run recovery, T4.5), while `GET /api/runs/:runId/lineage` returns the **persisted decided** rows (audit) — they are not the same list and neither is a substitute for the other. The lineage writes rows ordered by `created_at`/`started_at` with `id` as a tiebreak, which makes reads *stable* but does NOT guarantee ties resolve into declaration order — assert DQ results by name (lesson above), never by index.
- Adding jsdom tests raises the parallel load that tips latent races in OTHER pages into ~1-in-10 flakes — this has now happened twice (T4.5 tipped Review; T5.5 tipped Plan's rules upload). When it does, **do not** re-run until green and do not just raise the timeout: reproduce it deterministically first. Two tools that work — run the web suite 8× concurrently (`for i in $(seq 1 8); do npx vitest run web/src & done; wait`) to force it, and stash your changes to stress the baseline, which tells you whether you introduced the bug or merely exposed it. Then gate the suspect fetch behind a promise you release manually; if the assertion fails with the *full* findBy timeout (~1000ms) rather than appearing late, it is a real lost-update race, not a slow test. Plan's was real: the mount effect's `GET /rules` and a save/upload ran concurrently and whichever resolved LAST won, so a late read's `setCurrent(null)` wiped the just-uploaded artifact and the "Rules saved" confirmation with it. The page-level fix is that a **write must invalidate a read already in flight** — an effect-local `let active` only covers project-switch/unmount, so keep the load's cancellation token on a `useRef` too and flip it at the top of every write handler. Fix it in its own commit, with a gated-fetch regression test, before the feature commit.
- `platform.served_tables` is keyed by served **name** while the wizard carries a **project**, so any project→endpoint UI needs the explicit join `listServedTables(store,projectId)` + `GET /api/projects/:id/served` (T5.4) — there is no name on the project. A project with no row there has simply not published yet: that's a normal pre-run state to render (link back to Run), never a 404. Generic UI over a **generated** report must not assume column names: pick the measure/dimension by DuckDB type (first numeric / first non-numeric) and withhold rather than mislead — bars from a zero baseline hide the sign of a negative measure, a total summarized from one page must say "(page)", and a cell that arrived as a string because it's a BIGINT past JSON's safe range (T5.3 `toJsonCell`) must contribute 0 to a sum, never a coerced-rounded number.
- The PRD §5 acceptance test (T6.2, `tests/acceptance.test.ts`) runs the whole real path but **replays the model call** from committed cassettes (`tests/fixtures/model-responses/*.json`, loaded by `tests/helpers/model-cassette.ts`) captured from a live `opencode/big-pickle` run — so the executed SQL is genuinely the free model's, while `npm test` never depends on a free model's mood. That is not paranoia: one live attempt 422'd on malformed model JSON while 3/3 probes right after were valid. `ACCEPTANCE_LIVE_MODEL=1 npx vitest run tests/acceptance.test.ts` re-records the cassettes and proves criterion 6 by execution (it passed all 6 criteria live in 87s) — re-record whenever a prompt in `core/{plan,transform,dq}.ts` changes, since a stale cassette would test a prompt that no longer exists. Prove "nothing ran unapproved" by reading `GET /runs/:id/lineage` **before** posting each approval and asserting the tool has no call row yet — never by comparing timestamps, because the approvals route resolves the gate *before* writing the audit row, so approval.createdAt vs toolCall.startedAt genuinely races.
- **PRD §5 is a platform contract, not a report-correctness contract — the free model's transform is plausible-but-WRONG on the fixture and acceptance passes, correctly.** big-pickle emits `TRY_CAST(loan_amount AS DOUBLE)` on the thousands-separated text T6.1 built to trap exactly that (all 24 → NULL, so `overdue_amount` is 0 in every served row vs the true 117501.00), and dedups latest-per-`customer_id`, dropping C1003's second legitimate loan (active 9 vs 10). Nothing catches it: `DQ_TARGET_TABLE` is pinned to `raw.source`, the pipeline **INPUT**, so the generated checks never validate the published `marts` output at all. FR8's human SQL review is the designed answer (the model even surfaced it as an FR6 clarifying question). So do NOT "fix" the acceptance test by asserting the model's arithmetic, and do NOT read a green §5 as "the report is right" — if you want DQ over the published output, that is a product decision for parker, not a silent widening of the contract.
- The committed demo fixture (T6.1, `fixtures/loans_sample.csv`) has **load-bearing properties, not incidental ones** — change it and the acceptance flow silently degrades. `loan_amount` is thousands-separated TEXT (`"12,500.00"` → `read_csv_auto` infers VARCHAR; 0 of 24 values `TRY_CAST` to DOUBLE as-is) *on purpose*: make it numeric and rules.txt's "convert loan_amount to numeric" becomes a no-op the pipeline can pass without doing the work. It is deliberately NULL-FREE so the generated DQ checks PASS and the run reaches a served output — a null anywhere risks a generated `not_null` check failing and blocking publish, which would fail PRD §5 acceptance (the DQ-failure path keeps its own failing data in the DQ tool/runner tests; don't source it from the fixture). `candidateKeys` is `[]` and that is CORRECT, not a bug to fix: 24 rows hold 22 customers (one exact-duplicate row + one customer with two legitimately distinct loans), which is what makes "remove duplicate customers by customer_id" the genuine FR6 ambiguity — so never assert a non-empty candidate key on it. The overdue/active/closed buckets are all non-empty AND disjoint (no row is both `dpd_days > 0` and `balance = 0`), because the rules list them as UNORDERED bullets — an overlapping row would make the report depend on the model's CASE-clause order and turn a correct run into a flaky assertion. `created_at` must stay DATE-inferring or the generated freshness check has no column to bind.
- FR11's "paid providers authenticate from env keys" needs **no `client.auth.set(...)`** — ARCHITECTURE §8 sketches one, but OpenCode discovers a provider from its env key at boot and it simply appears in `config.providers()` with `source: "env"`. Verified live: with no key the catalog is 6 free `opencode/*` models (`source: "custom"`); with a **dummy** `ANTHROPIC_API_KEY` it also carries 15 paid `anthropic/*` models. So a paid model either exists in the catalog or genuinely does not — the T6.3 quality tier is therefore *really* empty without a key, and the picker names the key to set rather than offering a model that would fail at prompt time. `ModelInfo.cost` is USD **per 1M tokens** (`claude-sonnet-4-5` → `3`/`15` = its published $3/$15), not "per token" as core/types.ts's comment claimed until T6.3 fixed it. Remember the model only does work in the three generation stages (plan/transform/dq): the runner is deterministic (#48), so `run.model` is **provenance** — which model produced the artifacts a run executes — not a setting that changes the run. A `null` selection must OMIT `model` from the request so the backend's default applies; never duplicate the platform default into the UI.
- The 8×-parallel web stress from #56 can **false-positive**: it flagged `Serve.test.tsx` (~1-2/8), but stashing showed the SAME test fails on the clean baseline, so the stress had exposed nothing new. Diagnose before "fixing", and read the dumped DOM: it showed the project select populated yet neither the endpoints region NOR the "not published yet" message, i.e. `loaded` was still false — the fetch promise had never **settled**, where a lost update would have settled and been ignored. The discriminator is whether the effect's `active` flag *can* be false: it only flips on a dep change or unmount, and `projectId` only ever goes `""→"p-1"` once, so it cannot — therefore CPU starvation under artificial load, not a race. (`Serve.test.tsx` alone passes 10/10 at 10× parallelism; `npm test` is green.) Do not raise a timeout or invent a fix for this.
- Smoke-rendering a page against a **real backend** (the #54 recipe): the OpenCode SDK shares `globalThis.fetch`, so a Vite-proxy wrapper must rewrite **only string inputs starting with `/`** and pass everything else through untouched — the SDK calls fetch with a `Request` object, and `String(input)` on it yields `"[object Request]"`, which becomes a bogus URL and makes `GET /api/models` 500 while a direct `listModels(client)` call succeeds (that split is the tell). Also: any smoke that throws before its `close()` leaves `opencode serve` holding port 4096, and the next boot dies with an opaque `ServeError: Server exited with code 1` — `pkill -f "opencode serve"` between runs, and prefer a `try/finally`.
- **Docs are the one artifact with no gate, so they rot silently** — the README claimed "design phase, nothing built yet" through all 30 prior tasks and nothing ever failed. `tests/docs.test.ts` (T6.4) now pins README/DEMO claims to the code: documented npm scripts exist, documented routes are registered **and** every registered route is documented (both directions — a new route without a README row fails), links/fixture paths resolve, cited ports match `index.ts`/`vite.config.ts`. Enumerating routes needs care: `app.addHook("onRoute")` only fires for routes registered *after* the hook, so it CANNOT retro-collect from a built `buildServer()` (it silently yields an empty set — the vacuous-pass trap; guard every scraped set with a non-empty size assertion). `app.printRoutes({commonPrefix:false})` does list them, but the tree is **radix-compressed** — a bare `s` child under `/:id/source` means `/api/projects/:id/sources` — so a full URL is the concatenation of its ancestors' segments, and Fastify auto-adds `HEAD` to every `GET`. Reconstruct from the tree, then confirm each recovered URL with `app.hasRoute({method,url})` (exact pattern match, method-sensitive, `false` for a bogus path) so the check rests on the router, not on a debug string format. Grep-based route extraction is NOT a substitute: routes registered as `app.post<{Params:…}>(` or with the path on the next line evade the obvious regex (this repo has both — a naive grep found 4 of 23).

- `run_query` (V3.3) enforces read-only by SQL SHAPE, not a keyword denylist: a single statement beginning with SELECT/WITH cannot mutate in DuckDB — DuckDB has NO data-modifying CTEs (unlike Postgres `WITH ... INSERT`) and every write/DDL/ATTACH/COPY/PRAGMA starts with its own keyword — so the guard is (strip string literals + comments) → reject any `;` beyond a single trailing one → require a SELECT/WITH head. A denylist would false-positive on a column like `update_time`, a legit `OFFSET`, or a string literal `'please delete me'`; the shape check doesn't. Sources are queried BY NAME (FR5b) by exposing each session CSV as a `main."<name>"` view — but a bound `?` param CANNOT go inside `CREATE VIEW` (DuckDB: "Unexpected prepared parameter — this type of statement can't be prepared"), DDL can't be prepared, so the backend-controlled path is single-quote-escaped into the `read_csv_auto('...')` literal (same as the land/serve COPY `TO` destinations), NOT bound. Those views are DROPped in a `finally` after each query: the store is ONE shared DuckDB connection (single-user localhost), so a view left in `main` would let a later query — even another session — resolve a bare source name it never connected; create-per-call + teardown keeps each query scoped to its own session's sources.
